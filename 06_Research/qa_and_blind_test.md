## 一、QA 总体策略与流程

### 1.1 QA 目标

QA 贯穿《奇点回响》全流程，目标是确保最终成片在角色一致性、画面质量、叙事节奏、音频体验与技术合规五个维度达到"电影级质感"标准，并满足项目计划书定义的成功标准。

| 维度 | 量化目标 | 验证方式 |
|------|----------|----------|
| 角色一致性 | 跨镜头相似度 >95%；盲测平均分 ≥4.5 | 角色一致性盲测 |
| 视频质量 | 无明显闪烁、穿模、手指融合、首帧偏离 | 视频质量逐镜检查 |
| 叙事流畅度 | 连续观看无逻辑跳跃，节奏张弛有度 | 粗剪/精剪/成片三级评审 |
| 音频质量 | 对白清晰、配乐匹配、音效完整、混音平衡 | 音频质量检查 |
| 技术合规 | 4K / 24fps / −14 LUFS / 格式合规 | 技术合规自动+人工复核 |

### 1.2 QA 角色与职责

| 角色 | 职责 | 出席评审 |
|------|------|----------|
| 质量总监/QA | 制定标准、组织盲测、汇总报告、跟踪 Bug 闭环 | 全部 |
| 导演 | 叙事、节奏、镜头取舍最终判定 | 全部 |
| 美术指导 | 角色一致性、画面风格判定 | 第一/二阶段 |
| 后期总监 | 画质、修复、调色、混音技术判定 | 第三/四阶段 |
| 声音设计师 | 配音/配乐/音效/混音判定 | 第三/四阶段 |
| 教师/导师 | 阶段里程碑验收签字 | 各阶段末 |

### 1.3 QA 流程总览

```
资产/片段产出 ──→ 自检（责任人）──→ QA 抽检 ──→ 评审会 ──→ 通过？
                                                                  │
                                                          ┌───────┴───────┐
                                                          ▼               ▼
                                                        归档          返工/重生成
                                                          │               │
                                                          ▼               ▼
                                                     下一阶段      记录至失败案例表
```

### 1.4 QA 检查点（与项目里程碑对齐）

| 检查点 | 时间 | 检查内容 | 产出 |
|--------|------|----------|------|
| CP-1 | 第 2 周末 | 角色资产与关键帧评审 | 角色一致性盲测报告 |
| CP-2 | 第 4 周末 | 原始镜头质量抽检 | 视频质量抽检报告 |
| CP-3 | 第 5 周末 | 粗剪 + 音效审看 | 粗剪/精剪评审表 |
| CP-4 | 第 6 周末 | 最终成片与资产归档验收 | 成片 QA 报告 + 教师签字 |

### 1.5 QA 评审机制

- **创意评审会**：每周 2 次，导演/美术/后期/声音参加，审看关键帧、镜头、粗剪。
- **QA 验收会**：每阶段 1 次，QA + 教师 + PM 参加，对阶段交付物签字。
- **师生同步会**：每周 1 次，QA 汇总本周 Bug 与失败案例，通报修复情况。

## 二、角色一致性盲测方案

### 2.1 目标

验证 Flux.1 Kontext + IPAdapter 角色锁定管线产出的角色在不同镜头、不同场景、不同角度下保持身份一致，跨镜头相似度 >95%。

### 2.2 盲测人员

盲测评审组至少 4 人，覆盖创意、技术、教学三方视角：

| 序号 | 角色 | 视角 | 必到 |
|------|------|------|------|
| 1 | 导演 | 叙事与人物认同 | 是 |
| 2 | 美术指导 | 面部/服装/发型专业判断 | 是 |
| 3 | 质量总监/QA | 客观评分与流程合规 | 是 |
| 4 | 教师/导师 | 外部独立视角 | 是 |
| 5 | AI 算法工程师（可选） | 技术成因分析 | 否 |
| 6 | 后期总监（可选） | 后期可修补性判断 | 否 |

> 评审者不得提前接触待评图片的镜头编号与生成参数，确保"盲"测。

### 2.3 盲测方法

1. **样本准备**：从 24 镜头关键帧及各阶段视频截帧中，抽取同一角色（艾娃/核心）的图片，覆盖正面、侧面、远景、特写、不同场景（废墟/飞船/核心舱）。
2. **打乱编号**：QA 将所有图片随机打乱，赋予匿名编号（如 `BL_A001`、`BL_A002`...），不暴露镜头号、场景、生成批次。
3. **配对判断**：每次向评审者展示 2 张图片，询问"是否为同一人"。每位评审者独立判断，不互相讨论。
4. **填写记录表**：评审者按评分标准打分并填写盲测记录表。
5. **统计汇总**：QA 收集所有记录表，计算平均分、最低分、相似度百分比。

> 建议每组配对至少由 4 位评审者独立打分；同一角色的图片应形成多组配对，覆盖关键跨场景组合。

### 2.4 评分标准

| 分值 | 含义 | 判定描述 |
|------|------|----------|
| 5 | 完全是同一人 | 面部特征、发型、服装锚点高度一致，无任何违和 |
| 4 | 基本是同一人 | 整体一致，仅有极细微差异（如光照引起的肤色微变） |
| 3 | 相似但有差异 | 能看出是同一角色设定，但存在明显差异（五官/发型/服装局部变化） |
| 2 | 难以判定 | 似是而非，无法确认是否同一人 |
| 1 | 完全不同的人 | 面部特征、身份明显不同，观众会认为是两个人 |

### 2.5 通过标准

盲测通过需同时满足以下全部条件：

- **平均分 ≥ 4.5**（4 位评审者打分算术平均）
- **无人评分 ≤ 3**（任一评审者给 ≤3 分即视为该配对不通过）
- **相似度 > 95%**（以"评分 ≥4 的配对数 / 总配对数"估算）
- **无 P0 级角色崩坏**（如面部全黑、五官错位、性别突变）

> 任一条件不满足，该批次角色资产判定为不通过，需回到 Flux 工作流调整 IPAdapter 权重/参考图/提示词后重新生成，并记录至 `06_Research/failure_cases.md`。

## 三、视频质量检查标准

### 3.1 检查维度

对每个原始视频片段（Wan2.2 / 可灵生成）及后期各版本逐镜检查以下四项：

| 维度 | 检查内容 | 工具/方法 |
|------|----------|-----------|
| 闪烁检测 | 帧间亮度/色彩是否跳变 | 逐帧播放、波形图/矢量图监测 |
| 物理逻辑 | 手指数量、肢体结构、物体穿模、重力方向 | 逐帧人工审看 |
| 运动流畅度 | 主体运动是否卡顿/抽搐/反向，镜头运动是否自然 | 正常速度 + 慢速回放 |
| 首帧保真度 | 视频首帧与关键帧参考图是否一致 | 首帧截图与参考图 A/B 对比 |

### 3.2 闪烁检测（帧间一致性）

| 等级 | 表现 | 判定 |
|------|------|------|
| A | 无可见闪烁，帧间过渡平滑 | 通过 |
| B | 偶有轻微亮度波动，不影响观看 | 通过（记录） |
| C | 明显闪烁，每秒可见跳变 | 不通过，需 Low Noise Expert 修复或重生成 |
| D | 严重闪烁，画面"抖动"无法观看 | 不通过，必须重生成 |

**检测方法**：
- 在 DaVinci Resolve 中加载片段，打开波形图（Waveform）与矢量图（Vectorscope），观察是否存在帧间大幅跳变。
- 以 0.25× 慢速逐帧回放，人工确认无明显亮度/色彩突变。

### 3.3 物理逻辑检查

| 检查项 | 不通过表现 | 处理建议 |
|--------|------------|----------|
| 手指 | 多指、少指、融合、扭曲 | AE 逐帧修补或重生成 |
| 肢体 | 关节反折、肢体穿身、多余肢体 | 切分镜头或可灵首尾帧约束 |
| 物体穿模 | 手穿物体、物体穿墙、相互穿透 | 遮罩替换或重生成 |
| 重力/方向 | 人物悬浮、反向行走、物体乱飞 | 修正提示词方向描述后重生成 |
| 服装 | 服装瞬变、布料穿体、配饰消失 | 强化服装锚点提示词后重生成 |

### 3.4 运动流畅度

| 等级 | 表现 | 判定 |
|------|------|------|
| A | 运动自然流畅，符合物理直觉 | 通过 |
| B | 轻微卡顿，不影响叙事 | 通过（记录） |
| C | 明显卡顿/抽搐，主体运动不连贯 | 不通过，需修复或重生成 |
| D | 运动崩坏，主体形变或瞬移 | 不通过，必须重生成 |

### 3.5 首帧保真度

视频首帧应与 Flux 生成的关键帧参考图高度一致（用于图生视频 I2V）。

| 等级 | 表现 | 判定 |
|------|------|------|
| A | 首帧与参考图几乎完全一致 | 通过 |
| B | 构图/主体一致，细节有轻微偏移 | 通过（记录） |
| C | 主体存在但姿态/服装有明显偏移 | 不通过，降低 Denoise 至 0.5–0.6 重生成 |
| D | 首帧完全偏离参考图 | 不通过，使用 img2img 锁定或重生成 |

## 五、音频质量检查

### 5.1 检查维度

依据 `04_SOP/audio_production.md`，音频质量分四项检查：

## 八、Bug 分级与处理流程（P0–P3）

### 8.1 Bug 分级标准

| 等级 | 定义 | 示例 | 处理时限 | 处理方式 |
|------|------|------|----------|----------|
| **P0** | 阻断性严重问题，直接破坏观众认同或无法发布 | 角色面部全镜段不一致、手指明显畸形、性别突变、成片无法播放、响度严重超标削波 | 立即 | 必须修复或重生成，阻断进入下一阶段 |
| **P1** | 重大问题，明显影响沉浸感或合规 | 持续闪烁、明显穿帮、肢体穿模、对白不可懂、分辨率未达 4K、响度偏差 >2 LUFS | 当阶段内 | 应修复，修复后方可签字 |
| **P2** | 一般问题，可察觉但不致命 | 边缘伪影、轻微变形、配乐情绪略偏、转场略生硬、一闪而过的瑕疵（<3 帧） | 视成本决定 | 评估工时后决定修复或保留，记录在案 |
| **P3** | 轻微问题，观众不易察觉 | 极轻微色彩偏移、个别帧噪点、音效音量略低 | 可忽略 | 可保留，记录在案供后续迭代 |

### 8.2 Bug 处理流程

```
发现 Bug ──→ QA 登记到 Bug 清单（定级 P0–P3）──→ 分派责任人
                                                        │
                              ┌──────────────────────────┼──────────────────────────┐
                              ▼                          ▼                          ▼
                           P0/P1                       P2                         P3
                       立即/当阶段修复            评估工时决定               记录在案
                              │                          │                          │
                              ▼                          ▼                          ▼
                       修复/重生成                修复或保留                  归档
                              │                          │
                              ▼                          ▼
                       QA 复检 ──→ 通过？ ──→ 是 ──→ 关闭 Bug
                                  │
                                  否 ──→ 返回修复（升级定级若反复失败）
```

### 8.3 修复 vs 重生成决策

依据 `04_SOP/postproduction.md` 第 5.4 节，出现以下情况应放弃修复、回到生成工序重生成：

| 判定条件 | 处理 |
|----------|------|
| 单镜修复工时 > 重新生成工时 × 1.5 | 重生成 |
| 角色面部全镜段不一致（非局部） | 重生成，加强 IPAdapter 锚点 |
| 手指/肢体变形贯穿整镜 | 重生成，切分镜头或换可灵首尾帧 |
| 闪烁无法用时域平滑消除 | 重生成，降低 Wan2.2 Denoise |
| 修复后画面"塑料感"明显 | 重生成 |

> 所有"放弃修复改为重生成"的决策须记录至 `06_Research/failure_cases.md`，注明镜头号、问题、原因、重生成参数。
